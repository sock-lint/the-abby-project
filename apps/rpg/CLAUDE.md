# apps/rpg/

Habitica-inspired RPG layer: character profiles, streaks, drops, consumables (14 effects), cosmetics, the central `GameLoopService` orchestrator, and the runtime-authored sprite catalog. **The single entry point into this subsystem is `GameLoopService.on_task_completed(user, trigger_type, context)`** — every task-completion flow in the codebase routes through it.

## Models
- `CharacterProfile` (level, streaks, cosmetic slots, `streak_freeze_expires_at`, boost timers, `active_trophy_badge`)
- `ItemDefinition` (10 item types incl. consumable, 5 rarities). Typed cross-entity FKs: `pet_species`, `potion_type`, `food_species`. `metadata` JSONField for free-form per-type fields.
- `UserInventory`, `DropTable`, `DropLog`
- `SpriteAsset` (runtime-authored sprite catalog — slug + ImageField + animation metadata + authoring inputs `prompt / original_intent / motion / style_hint / tile_size / reference_image_url` — persisted so reroll can replay them).

## Services (`apps/rpg/services.py`)
- `StreakService`, `DropService`, `CosmeticService`, `ConsumableService`, `GameLoopService` (central orchestrator).
- `sprite_authoring.py`: `register_sprite`, `register_sprite_batch`, `delete_sprite`, `get_catalog` — write to the abby-sprites Ceph bucket.
- `storage.py::sprite_storage()` — lazy-helper for the dedicated sprites STORAGES key.
- `constants.py`: shared `TriggerType` TextChoices.

Celery task: `evaluate_perfect_day_task`.

## Endpoints
- `/api/sprites/catalog/` (`SpriteCatalogView`, public, ETag-cached) — read API.
- Parent-only admin CRUD at `/api/sprites/admin/*` in `sprite_admin_views.py` powers the `/manage → Codex → Sprites` panel.
- `POST /api/inventory/<item_id>/use/` — consume one consumable.
- `POST /api/character/trophy/` — set/clear the active trophy badge.
- `GET /api/cosmetics/` — owned cosmetics grouped by slot.
- `GET /api/cosmetics/catalog/` (child-accessible) — every authored cosmetic grouped by slot keys; non-cosmetic items filtered out server-side via `COSMETIC_SLOT_MAP.keys()`.

## Gotchas

- **RPG game loop**: `GameLoopService.on_task_completed(user, trigger_type, context)` is the single orchestrator called from all existing flows — clock-out (`apps/timecards/services.py`), chore approval (`apps/chores/services.py`), project/milestone completion signals (`apps/projects/signals.py`), homework create + approval (`apps/homework/services.py`), habit taps (`apps/rpg/views.py`), journal first-of-day (`apps/chronicle/services.py`), and creation first-2-per-day (`apps/creations/services.py`). Pipeline: (1) `StreakService.record_activity` updates `CharacterProfile.login_streak`, first-of-day action awards streak-scaled check-in coins (`5 × min(1 + streak × 0.10, 3.0)` — 5c at day 1, 15c cap at day 20+) via `CoinLedger.Reason.ADJUSTMENT`; (2) `DropService.process_drops` rolls against `BASE_DROP_RATES[trigger_type]` + streak bonus (`min(streak × 0.05, 0.50)`) — callers can set `context["drops_allowed"]=False` to suppress the drop roll while still recording streak + quest progress (used by `homework_created` to cap drops to the first log per day); (3) `QuestService.record_progress` applies damage/collection if the user has an active quest and the trigger matches the quest's `trigger_filter`. Returns `{trigger_type, streak, drops, quest, notifications}` — failures in quest progress are logged but never break the parent flow (wrapped in `try/except`). **`trigger_type` must be a value from `apps.rpg.constants.TriggerType`** — the same enum that backs `DropTable.TriggerType`, `BASE_DROP_RATES` (rpg/services), and `TRIGGER_DAMAGE` (quests/services). The content-pack loader validates `drops.yaml` triggers against it, so a typo raises `ContentPackError` before any write commits.

- **Streaks & daily engagement**: `CharacterProfile` auto-created per user via `post_save` signal on `accounts.User` (see `apps/rpg/signals.py`). `login_streak` resets to 1 on a gap > 1 day, increments on consecutive days, and is tracked alongside `longest_login_streak`. Milestone notifications fire at streaks of 3, 7, 14, 30, 60, 100 (`NotificationType.STREAK_MILESTONE`). **Perfect Day:** `evaluate_perfect_day_task` (Celery, 23:55 local) awards `perfect_days_count += 1` and 15 bonus coins to any child who was active today AND has at least one daily chore scheduled AND completed every daily chore. The "at least one scheduled" guard (added 2026-04-26) prevents days with zero daily chores from auto-qualifying — pinned by [`apps/rpg/tests/test_tasks.py`](tests/test_tasks.py). **Streak Freeze:** `CharacterProfile.streak_freeze_expires_at` (DateField) is a one-shot grace set by consuming a `ItemType.CONSUMABLE` with `metadata.effect="streak_freeze"`. When `StreakService.record_activity` sees a gap > 1 day, it consumes the freeze if the expiry covers the first missed day — streak continues, field clears. Gentle-nudge design: missed days dim the streak flame visually and reset the multiplier, but NEVER destroy earned coins/XP/items.

- **Consumables**: `ItemType.CONSUMABLE` items fire a one-shot effect on use and decrement `UserInventory.quantity`. Dispatch lives in `ConsumableService._apply_effect` in `apps/rpg/services.py`, keyed by `metadata.effect` — unknown effects raise `ValueError` so a typo in YAML doesn't silently no-op. **14 effects ship today:** `streak_freeze` (protects one missed day; `metadata.duration_days`), `xp_boost` (Scholar's Draught — timer gate via `CharacterProfile.xp_boost_expires_at`; `metadata.duration_hours`), `coin_boost` (Lucky Coin — `coin_boost_expires_at`), `drop_boost` (Drop Charm — `drop_boost_expires_at`), `growth_tonic` (+2x growth for next N pet feeds; increments `CharacterProfile.pet_growth_boost_remaining`), `rage_breaker` (clears `Quest.rage_shield` on the user's active boss quest, erroring if none), `growth_surge` (Pet Evolution Accelerator — bolt of growth to active pet), `feast_platter` (+N growth to every unevolved pet), `mystery_box` (random common/uncommon non-cosmetic item), and **five added 2026-04-23**: `lucky_dip` (random uncommon-or-rarer cosmetic, auto-salvages dupes to coins), `quest_reroll` (spawns a random eligible system quest when user has no active quest), `morale_tonic` (+2x growth for next 5 feeds — longer-duration growth_tonic variant; distinct effect slug so `consumable_variety` counts it separately), `skill_tonic` (+N XP to the user's highest-level skill, promotes levels via `XP_THRESHOLDS` math), `food_basket` (N random food items). Boost multipliers are **fully wired** now: `xp_boost_multiplier(user)` in `apps/rpg/services.py` is called from `apps/achievements/services.py:275`; `coin_boost_multiplier(user)` from `apps/rewards/services.py:44` (via `is_boostable_coin_reason` whitelist — excludes refunds, redemptions, exchanges, and mixed-sign adjustments); `drop_boost_additive(user)` from `DropService.process_drops:258`. Timer gates + multipliers are pinned by `apps/rpg/tests/test_boost_multipliers.py`. Streak-freeze also increments `CharacterProfile.streak_freezes_used`, exposed as the `STREAK_FREEZE_USED` badge criterion. Endpoint: `POST /api/inventory/<item_id>/use/` accepts an optional `quantity` body field (default `1`); stack-safe effects (counters / random rolls / additive XP — `mystery_box`, `lucky_dip`, `food_basket`, `growth_tonic`, `morale_tonic`, `skill_tonic`, `feast_platter`) loop the apply path N times, while stack-unsafe effects (timer overwrites + single-target — `xp_boost`, `coin_boost`, `drop_boost`, `streak_freeze`, `rage_breaker`, `growth_surge`, `quest_reroll`) reject `N > 1` with 400 (the second use would clobber the first timer or no-op against an already-cleared shield). The classification lives in `_STACK_SAFE_EFFECTS` in `apps/rpg/services.py`. Adding a new consumable means (a) seed entry in `content/rpg/initial/items.yaml` with `item_type: consumable` + `metadata.effect: <slug>`, (b) new branch in `_apply_effect`, and (c) decide whether to add the slug to `_STACK_SAFE_EFFECTS`. **Active boost timers** are exposed on `CharacterProfileSerializer` (`xp_boost_expires_at` / `coin_boost_expires_at` / `drop_boost_expires_at` / `pet_growth_boost_remaining`) so the frontend [`BoostStrip`](/frontend/src/components/rpg/BoostStrip.jsx) can render `1h12m left` chips on Inventory + the Sigil Frontispiece — kids no longer have to guess whether their potion is still working. Frontend `Inventory.jsx` renders a "Use" button on any `item_type: consumable` card; cards with `stack > 1` AND a stack-safe effect get a `−/+/all` quantity stepper plus a `Use × N` label.

- **Drops & inventory**: `ItemDefinition` is the master catalog (10 item types: egg, potion, food, cosmetic_frame, cosmetic_title, cosmetic_theme, cosmetic_pet_accessory, quest_scroll, coin_pouch, consumable; 5 rarities). Cross-entity references are typed: `ItemDefinition.pet_species` FK (eggs → `PetSpecies`), `ItemDefinition.potion_type` FK (potions → `PotionType`), `ItemDefinition.food_species` FK (food → preferred `PetSpecies`). `metadata` JSONField stays for genuinely free-form per-type fields (border colors, title text, coin-pouch counts). `DropTable` maps trigger → item → weight with `min_level` gating. `DropService.process_drops` rolls `random.random()` against effective rate, weighted-selects from eligible rows, handles **cosmetic salvage** (already-owned cosmetics auto-convert to coins via `item.coin_value`), increments `UserInventory.quantity`, and logs to `DropLog`. Frontend `useDropToasts` polls `/api/drops/recent/` every 20s and emits rarity-colored toast celebrations via `DropToastStack` (mounted globally in Layout).

- **Cosmetics**: `CharacterProfile` has 4 nullable FK slots (`active_frame`, `active_title`, `active_theme`, `active_pet_accessory`) with `limit_choices_to={"item_type": ...}` enforcing correct type. `CosmeticService.equip(user, item_id)` validates ownership in `UserInventory` (cosmetics are NOT consumed on equip) and writes to the correct slot based on `COSMETIC_SLOT_MAP`. `CosmeticService.unequip(user, slot)` nils the slot. Duplicate cosmetic drops auto-salvage (see Drops gotcha). `GET /api/cosmetics/` returns owned cosmetics grouped by slot; `GET /api/cosmetics/catalog/` (added 2026-04-22, child-accessible) returns *every* authored cosmetic grouped by the same slot keys so the Frontispiece can render un-owned cosmetics as locked intaglios. Non-cosmetic items (eggs/potions/food/etc.) are filtered out server-side via `COSMETIC_SLOT_MAP.keys()`, so children can't use the catalog endpoint as a backdoor to `ItemDefinition` rows they shouldn't see.

- **Trophy shelf** (`CharacterProfile.active_trophy_badge`, added 2026-04-23; frontend picker wired 2026-04-22): optional "hero badge" FK the user elects to display on their profile. Endpoint `POST /api/character/trophy/ {badge_id}` — the server enforces that the user has **earned** the badge (via `UserBadge`) before setting, so parent-authored or admin-forced trophies can't bypass the "earn it first" rule. Pass `badge_id: null` to clear. Exposed on `CharacterProfileSerializer` as `active_trophy_badge` (compact `{id, name, icon, rarity, description}` shape). FK uses `on_delete=SET_NULL` so archiving badges doesn't break profiles. Picker UI lives in [`TrophyBadgePicker`](/frontend/src/pages/character/TrophyBadgePicker.jsx) — a `BottomSheet` that reuses the Reliquary Codex's `groupBadgesByCollection` filtered to earned-only, so the seven-chapter grouping at `/atlas?tab=badges` is the same grouping you see when choosing your hero seal.

- **Sprite generation from text** (`apps/rpg/sprite_generation.py`): The `generate_sprite_sheet` MCP tool (parent-only) calls Google's **Gemini 3 Pro Image** ("Nano Banana Pro", `gemini-3-pro-image-preview`) to produce pixel-art sprites from a text prompt. The `update_sprite_metadata` MCP tool (parent-only) edits `fps` or `pack` on an existing sprite without regenerating the image — cheap way to dial animation speed or recurate a pack.
  - **One-shot pose sheet architecture**: animated strips are generated via a SINGLE Gemini call that draws all N frames as one composition — NOT iterative. Giving the model the whole sequence in one image is what makes character design, palette, and scale consistent by construction; per-frame calls with reference-image passing could never hit that bar no matter how strongly we prompted.
  - **8 motion templates** (pick via the `motion` param, default `idle`): `idle` (subtle breathing/micro-motion, most forgiving for Gemini), `walk` (4-phase quadruped cycle), `bounce` (squash-and-stretch in place — NOT vertical displacement; the subject stays stationary and only its proportions deform), `bubble` (liquid swirl inside a container — potions/cauldrons), `flicker` (flame oscillation), `glow` (pulsing halo — chests, magic items), `wobble` (gentle rock side-to-side — eggs about to hatch), `sway` (plants, flags). Each template is a 4-phase cyclic sequence where frame 4 leads naturally back to frame 1 for seamless looping.
  - **Full pipeline per call** — steps in order: `_fetch_reference_image` (if URL provided) → optional `_tighten_reference_to_subject` (reroll path only — autocrops the reference's existing 90% pipeline-padding to ~3% margin so Gemini doesn't compound the framing across rerolls) → `_preprocess_reference_image` (composite onto magenta so Gemini doesn't copy a white background from the reference's implied transparency) → `_generate_frame` (sole Gemini API seam; tests mock this so `google-genai` isn't required at test time) → `_chroma_key_to_transparent` (removes the magenta background Gemini was instructed to paint) → **STATIC PATH**: `_keep_largest_component` (prune orphan specks first — see "Subject sizing pipeline-order" below) → `_autocenter_frame` (scale to fit inner 90% window of `tile_size`). **ANIMATED PATH**: `_extract_frames_in_reading_order` (connected-components → take N largest → sort by row-band then left-to-right; handles ANY layout — 1×N horizontal, 2×2 grid, N×1 vertical — not just horizontal strips) → per-tile ground-align (subjects' bottoms anchored to a shared baseline so natural body bob reads as animation instead of getting normalized away) → `_keep_largest_component` per tile (scrubs chroma-key anti-alias fringe orphans that would otherwise flash as "cut-off portions from another frame") → `sprite_authoring.register_sprite` (Ceph upload + DB row via the already-tested write path) → `_frame_occupancy` measurement (attaches `pipeline_warnings` to the result dict when the largest emitted frame fills less than `OCCUPANCY_WARNING_FLOOR` (default 70%) of the tile so the admin UI can surface a "consider reroll" chip; never auto-retries — every retry burns ~$0.04 × frame_count).
  - **Subject sizing pipeline-order** (the "subjects too small" review): the static path now runs `_keep_largest_component` BEFORE `_autocenter_frame`. Pre-fix the order was reversed, which meant stray scene-decoration specks (anti-alias fringes, sparks Gemini drew despite the suffix bans) inflated the bbox during autocenter — the actual subject scaled down to fit that inflated bbox into the inner window, ending up at ~50% of the tile while the orphan pruning that ran AFTER could only remove the orphans, not recover the lost size. Pinned by `test_static_path_prunes_orphans_before_autocenter` in [`apps/rpg/tests/test_sprite_generation.py`](tests/test_sprite_generation.py). The animated path was always correct here because `_extract_frames_in_reading_order` finds components first.
  - **Subject size constants**: `AUTOCENTER_PADDING_FRAC = 0.05` ([apps/rpg/sprite_generation.py:41](sprite_generation.py)) → subject's max dimension scales to 90% of the tile (was 80% pre-2026-05). `OCCUPANCY_WARNING_FLOOR = 0.70` → below this, the response carries a `pipeline_warnings` entry telling the admin UI / chat client to flag the sprite for reroll. Default `tile_size` is **128** (was 64) — the frontend renders most sprites at 32-48px, so the 4× downsample preserves detail without sharpness loss; bumping the default gave Gemini more pixels to work with for detail-rich subjects. Both constants are pinned by `PipelineConstantTests` so future "just bump it a little" PRs trip a test instead of silently undoing the tuning. The prompt suffix and pose-sheet prompt both include explicit OCCUPANCY clauses ("subject MUST fill at LEAST 85% of the canvas in its longest dimension") because Gemini's default scale prior leans conservative — telling it the target occupancy raises baseline output size noticeably.
  - **Reference images** (`reference_image_url`, optional): caller provides a URL; service passes it to Gemini as a style + character anchor. **HIGHLY prescriptive — Gemini over-copies creature class.** Rules of thumb: (a) for creatures Gemini already knows well (fox, cat, dog), OMIT the reference and rely on text — a bear-anchored "fox" will draw a bear. (b) for items / stylized content, anchor against `apple-d6446566.png` (a clean un-overwritten pack sprite) for style transfer without shape bias. (c) for self-animating an existing sprite, self-anchor via its current URL works great. The pre-composite-onto-magenta step in `_preprocess_reference_image` is critical — without it, Gemini uses the reference's display-rendered-white background as a style cue and the chroma-key does nothing.
  - **Debug artifacts** (`return_debug_raw=true`): saves the raw Gemini output AND the post-chroma-key intermediate to `rpg-sprites/debug/<slug>-<stage>.png` on Ceph and returns URLs in the response. Indispensable when diagnosing unexpected output — lets you see what Gemini actually drew vs what our pipeline did to it.
  - **Error semantics**: `SpriteGenerationError` is the single service-layer exception; the MCP tool wraps it as `MCPValidationError`. No partial writes — if any stage fails, no `SpriteAsset` row is created and no Ceph blob is uploaded.
  - **`frame_count` capped** by `settings.SPRITE_GENERATION_MAX_FRAMES` (default 8) to bound worst-case API spend per invocation.
  - **Gemini failure modes and their remedies** (worth knowing before iterating on a new subject): (1) draws a 2×2 grid instead of the requested horizontal strip — the layout-aware extractor handles this automatically. (2) hallucinates scene elements (pink panels, ground strips, motion lines) despite enumerated bans — re-roll with `overwrite=True`. (3) interprets "stack" / "pile" as "multiple small copies in a row" — the fix is "ONE single [subject] filling most of the frame, NOT multiple tiny [subjects] in a row" wording. (4) uses reference image's visible background as a style cue — `_preprocess_reference_image` pre-composites onto magenta to block this. (5) bounces visually subtle at 4 frames because the sequential-keyframe rule clashes with squash-stretch — the bounce template includes explicit "this rule does NOT apply to this motion" language as an override.
  - **Requires `GEMINI_API_KEY`**; empty key raises at call time with a clear message rather than 500ing.
  - **Authoring inputs persist for reroll** (2026-04-23): the service writes `prompt / original_intent / motion / style_hint / tile_size / reference_image_url` back onto the `SpriteAsset` row at the end of the pipeline via a single `SpriteAsset.objects.filter(slug=slug).update(...)` call. This is what makes a one-click reroll work — `POST /api/sprites/admin/<slug>/reroll/` just replays those stored inputs with `overwrite=True`. `register_sprite` (the non-generating path) intentionally leaves these fields empty, so legacy sprites and raw-PNG uploads have no stored prompt and the reroll endpoint returns 400 with "no stored prompt — use Replace instead". Pinned by `test_authoring_inputs_persist_for_reroll` in [`tests/test_sprite_generation.py`](tests/test_sprite_generation.py). `original_intent` (added 2026-05-10) holds the user's plain-English description separately from the refined Gemini-ready `prompt` so a chat-side meta-prompt loop can re-refine from the same starting intent on reroll instead of replaying the cached prompt verbatim — pinned by `test_original_intent_persists_for_reroll` in the same file.
  - **Chat-side meta-prompt loop** (extended 2026-05-11): four MCP tools now support a chat-native refinement workflow without REST or frontend changes.
    1. `get_sprite_prompting_playbook` (parent-only, [`apps/mcp_server/tools/sprite_prompting_playbook.py`](/apps/mcp_server/tools/sprite_prompting_playbook.py)) returns a structured markdown document covering the failure modes, motion templates, subject/reference rules, tile/frame guidance, AND the "Diagnosing 'subject too small'" diagnosis checklist (added 2026-05-11). The motion section is composed from `MOTION_TEMPLATES` at module-load time so adding a new motion in `sprite_generation.py` automatically surfaces in the playbook.
    2. `get_sprite(slug)` (parent-only, sibling of `list_sprites` in [`apps/mcp_server/tools/sprite_authoring.py`](/apps/mcp_server/tools/sprite_authoring.py)) returns the full authoring shape including `prompt / original_intent / motion / style_hint / tile_size / reference_image_url` so Claude in chat can read what was sent to Gemini, fetch the rendered image at `url` for vision critique, and propose a reroll prompt. `list_sprites` stays lean (slug + url + structural metadata only) — the lookup is intentional, not bundled into every list call.
    3. **`propose_sprite_reroll(slug, critique, target_changes=[])`** (parent-only, 2026-05-11) — pure prompt-construction tool that does NOT call Gemini. Takes free-form critique text plus optional explicit refinement levers and returns the proposed refined prompt + motion/style_hint/tile_size overrides + per-lever rationale. Keyword detection maps common complaints ("too small", "wrong pose", "low detail", etc.) to lever slugs (`larger_subject` / `switch_motion` / `strengthen_style` / `drop_reference` / `bigger_tile`). The chat agent calls this AFTER fetching the rendered sprite and forming a vision-based critique; it then passes the proposed inputs to `reroll_sprite`. Unknown explicit levers raise `MCPValidationError` (typo-safety).
    4. **`reroll_sprite(slug, ...)`** (parent-only, extended 2026-05-11) — now accepts optional `prompt / motion / style_hint / tile_size` overrides on top of the stored row. The generation service's end-of-pipeline `update(...)` writes whatever inputs it ran with back onto the row, so a subsequent reroll resumes from the refined state automatically (no separate persist step). The reroll path also passes `tighten_reference=True` to the generation service so a self-anchored reroll (passing the existing sprite's PNG as the reference) doesn't compound the prior framing across iterations.
    Tests in [`apps/mcp_server/tests/test_sprite_prompting_playbook.py`](/apps/mcp_server/tests/test_sprite_prompting_playbook.py) and the `RerollSpriteToolTests` + `ProposeSpriteRerollToolTests` classes in [`apps/mcp_server/tests/test_tools_sprite_authoring.py`](/apps/mcp_server/tests/test_tools_sprite_authoring.py) pin the surface.
  - **Canonical CSS contract** (enforced in production [`RpgSprite.jsx`](/frontend/src/components/rpg/RpgSprite.jsx)): animated strips use `steps(N)` (= jump-end) with `endX = -N × size`. Do NOT use `steps(N, jump-none)` with `endX = -(N-1) × size` — it holds only the first N-1 frames for 1/(N-1) duration each and flashes frame N-1 briefly at the loop boundary (users perceive this as a bleed/ghost artifact). The difference is pinned by [`RpgSprite.test.jsx`](/frontend/src/components/rpg/RpgSprite.test.jsx).

- **Sprite admin UI on `/manage → Codex`** (`frontend/src/pages/manage/CodexSection.jsx` + sibling `Sprites*` / `Mounts*` files): the Codex's content is split across five sub-sections — Items / Creatures / Mounts / Adventures / Sprites. Mounts is a read-only browse of every `PetSpecies` rendered with its `{sprite_key}-mount` sprite (fallback to the base creature sprite via `CatalogCard.fallbackSpriteKey`); clicking a tile opens a BottomSheet with a "Regenerate mount sprite" button that hands off to `SpriteGenerateModal` in `replace` mode with the slug pre-filled. Sprites is the master authoring surface — a pack-grouped grid of every `SpriteAsset` with a top-right "Create sprite" button + slug search. Clicking a sprite opens [`SpriteDetailSheet`](/frontend/src/pages/manage/SpriteDetailSheet.jsx) which exposes four actions: **Reroll** (primary; disabled with tooltip until a stored prompt exists; gated by a `ConfirmDialog` showing estimated Gemini cost `frame_count × $0.04` — no server-side rate limit, the dialog is the only guardrail), **Replace** (opens `SpriteGenerateModal` in replace mode pre-filled from the current row), **Edit metadata** (fps + pack only — routes through `PATCH /api/sprites/admin/<slug>/`), **Delete** (confirm + blob-first). The five endpoints sit on [`apps/rpg/sprite_admin_views.py`](sprite_admin_views.py), all `permission_classes = [IsAuthenticated, IsParent]`:
  - `GET /api/sprites/admin/` — full catalog with authoring inputs (parent-only; different shape from the public `/api/sprites/catalog/` which stays lean + auth-less)
  - `POST /api/sprites/admin/generate/` — thin wrapper around `generate_sprite_sheet(actor=request.user, …)`
  - `POST /api/sprites/admin/<slug>/reroll/` — loads the sprite, 400s with `{detail: "no stored prompt — use Replace instead"}` if `prompt=""`, otherwise calls the service with stored inputs + `overwrite=True`
  - `PATCH /api/sprites/admin/<slug>/` — fps / pack edits via `update_sprite_metadata`
  - `DELETE /api/sprites/admin/<slug>/` — blob-first via `delete_sprite`
  After any write, the frontend calls `SpriteCatalogProvider.refetchCatalog({ bypassEtag: true })` so the new image shows up across every `<RpgSprite>` in the app without a page reload — the catalog response's ETag would otherwise return 304 and serve the stale cached map. The `refetchCatalog` field is on the provider's context value; the default context provides a no-op so components that mount outside the provider (e.g. isolated tests) don't crash.

- **Runtime sprite authoring**: Sprites used by `RpgSprite.jsx` are served from the `abby-sprites` Ceph bucket rather than bundled into the Vite build. The authoritative DB record is `SpriteAsset` (slug, ImageField, `pack`, `frame_count`, `fps`, `frame_width_px`, `frame_height_px`, `frame_layout`). Static sprites have `frame_count=1` and `fps=0`; animated sprites have `frame_count>1` and `fps≥1` — `SpriteAsset.clean()` enforces these invariants so invalid rows can't be saved. The bucket is configured `public-read` with `querystring_auth: False` (plain permanent URLs, no presigning) and `file_overwrite: True`, which is safe because `sprite_authoring.py` names every file `<slug>-<sha256[:8]>.<ext>` — the content hash makes collisions impossible and allows safe overwrites. `Cache-Control: public, max-age=31536000, immutable` is stamped on upload so browsers cache forever. The bucket is separate from the user-media bucket (`STORAGES["default"]`) — never mix the two; user media is private+presigned, sprites are public+permanent. The six MCP tools in this subsystem (`register_sprite`, `register_sprite_batch`, `list_sprites`, `delete_sprite`, `update_sprite_metadata`, `generate_sprite_sheet` — see the "Sprite generation from text" gotcha for the last one) are parent-only via `require_parent()` and live in `apps/mcp_server/tools/sprite_authoring.py`. `delete_sprite` follows the blob-first invariant (same pattern as `ProjectPhotoViewSet.destroy`). The frontend `SpriteCatalogProvider` (mounted in `App.jsx`) fetches `/api/sprites/catalog/` at boot, caches the response in `localStorage` under `spriteCatalog` + `spriteCatalogEtag`, and revalidates with `If-None-Match` → 304 on subsequent mounts. The catalog response includes animation metadata; when any animated sprite is present the provider emits a `<style id="sprite-keyframes">` tag with a single `@keyframes sprite-cycle` rule that all animated `RpgSprite` instances share via a CSS custom property (`--sprite-end-x`). A `prefers-reduced-motion` rule in `index.css` freezes the animation for users who opt out. `GET /api/sprites/catalog/` opts out of the blanket `Cache-Control: no-store` that `NoCacheAPIMiddleware` applies to all `/api/*` responses — it does this by setting its own `Cache-Control` header before the middleware runs; the middleware now checks and skips stamping if a header is already set. This is currently the **only** `/api/` endpoint that opts out of no-store. Any future endpoint that does the same must contain NO user-scoped data — the default no-store is the safety net for auth'd responses. The legacy `register_sprite_assets` tool (`apps/mcp_server/tools/sprite_assets.py`) still exists for local-dev source-tree authoring and is not deprecated; both pipelines coexist until a future Phase 3 cleanup. See `docs/superpowers/specs/2026-04-20-chat-to-prod-sprite-authoring-design.md` for the full architecture.

- **RPG content authoring** (`content/rpg/`): Pets, eggs, potions, items, drops, quests, badges, and the skill tree are authored as YAML in `content/rpg/initial/*.yaml` and loaded by `python manage.py loadrpgcontent` — a dependency-ordered, idempotent upsert. `seed_data` invokes this internally. The loader (`apps/rpg/content/loader.py`) is the single source of truth for cross-entity wiring: one `pet_species.yaml` entry fans out to both a `PetSpecies` row and an egg `ItemDefinition` with typed FK; same pattern for `potion_types.yaml` → potion items. Third-party packs live under `content/rpg/packs/<name>/` and are loaded with `--namespace <prefix>-` so slugs don't collide with core. When adding new RPG content, edit YAML — don't add `get_or_create` calls to Python seed code. See `content/rpg/README.md` for schema + authoring workflow.

## RPG tunables (hardcoded constants in `apps/rpg/services.py`)
These are intentionally NOT in settings — they're game-design constants, not deploy-time config.
- `BASE_CHECK_IN_COINS = 5` — base daily check-in bonus before streak multiplier.
- `STREAK_MULTIPLIER_PER_DAY = 0.10` and `STREAK_MULTIPLIER_CAP = 3.0` — check-in bonus scales `min(1 + streak × 0.10, 3.0)` (5c at day 1, 15c cap at day 20+). Bumped 2026-04-23 from 0.07 so the cap reaches at ~3 weeks instead of ~4.
- `BASE_DROP_RATES` — dict of `TriggerType` → base drop probability (clock_out 0.40, chore_complete 0.30, homework_complete 0.35, homework_created 0.15, milestone_complete 0.80, project_complete 1.00, badge_earned 1.00, quest_complete 1.00, perfect_day 1.00, habit_log 0.15, savings_goal_complete 0.80, journal_entry 0.20, creation_logged 0.25). `journal_entry` was bumped from 0.10 → 0.20 in the 2026-05 progression-UX pass — the DB unique constraint on `(user, occurred_on) where kind=journal` already throttles journals to one per local day, so a higher rate is fairness-rebalancing against `habit_log` (which can fire 4+ taps/day at 0.15) rather than a farm risk. Three triggers are daily-capped at the service layer: `homework_created` (first-of-day only via `HomeworkDailyCounter`), `journal_entry` (one per local day, DB-enforced via partial unique constraint on `ChronicleEntry`), and `creation_logged` (first 2 per local day via `CreationDailyCounter`). The two counter-backed gates share the `config.base_models.DailyCounterModel` abstract + `config.services.bump_daily_counter` helper. All three close streak/drop/quest farming at source — counter rows survive both soft-delete and hard-delete, so collusion-style resets (create → delete → create) don't re-arm.
- `STREAK_DROP_BONUS_PER_DAY = 0.05` and `STREAK_DROP_BONUS_CAP = 0.50` — streak adds up to +50% to base drop rate.
