# apps/quests/

Boss + collection quests with time-boxed end dates, badge gating, rage shield, co-op support, and a separate daily-challenge micro-quest slot. Quest progress is driven by `GameLoopService.on_task_completed` — never call quest services directly from feature flows.

## Models
- `QuestDefinition` (boss/collection, `trigger_filter` JSON, `required_badge` gating, `xp_reward`/`coin_reward`/`QuestRewardItem`).
- `Quest` (active quest row per user — `status`, `current_progress`, `rage_shield`, `end_date`).
- `QuestParticipant` (multi-player-ready, wired today for parent-authored co-op quests).
- `QuestRewardItem` (per-quest reward bundle).
- `QuestSkillTag` (XP fan-out — same shape as `ProjectSkillTag`).
- `DailyChallenge` (separate slot — unique on `(user, date)`; 5 challenge types).

## Services
- `QuestService` — `start`, `record_progress`, `_complete_quest`, `expire`, `apply_boss_rage`, `start_co_op_quest`.
- `DailyChallengeService` — `get_or_create_today`, `record_progress`, `claim_reward`.

Celery tasks: `expire_quests_task` (00:10 local), `apply_boss_rage_task` (00:15 local), `rotate_daily_challenges_task` (00:30 local).

## Endpoints
- `POST /api/quests/{definition_id}/co-op/ {coop_user_ids}` (parent-only, family-scoped) — wraps `QuestService.start_co_op_quest`.
- `GET /api/challenges/daily/` and `POST /api/challenges/daily/claim/`.

## Gotchas

- **Quests**: Two types — `boss` (damage-based HP pool) and `collection` (item-count target). `Quest` has a time-boxed `end_date` and tracks `current_progress` and `rage_shield` (extra damage required after an idle day). One active quest per user at a time (`Quest.status=="active"`). **Damage formula** per trigger in `TRIGGER_DAMAGE`: clock_out 10/hr, chore 15, homework_complete 25, homework_created 5, milestone 50, badge 30, project_complete 75, habit_log 5. Collection quests count 1 per qualifying trigger. **Trigger filtering** via `QuestDefinition.trigger_filter` JSONField: `allowed_triggers` (whitelist of `TriggerType` values), `project_id`, `skill_category_id`, `chore_ids`, `savings_goal_id`, `on_time` (boolean — when true, only counts homework contexts with `on_time=True`) — empty filter = all triggers count. Validated via `apps/quests/validators.py::validate_trigger_filter` — wired into `QuestDefinition.clean()`, `QuestWriteSerializer`, and MCP `create_quest_definition`; unknown keys or trigger values raise `ValidationError` before any write. The 2026-04-23 review removed `streak_target` and `perfect_day_target` from the allowlist — they were validated but never read, producing silently-broken quests. Re-add only when a reader lands in `QuestService.record_progress`. **Badge gating:** `QuestDefinition.required_badge` blocks start until earned. **Rage shield:** `apply_boss_rage_task` (Celery, 00:15 local) ticks every active boss quest: idle day climbs by `RAGE_SHIELD_CLIMB_STEP` (15) capped at `RAGE_SHIELD_CAP` (100); active day decays by `RAGE_SHIELD_DECAY_STEP` (25) toward 0. The decay-on-return path preserves the gentle-nudge doctrine — absence costs progress, but return clears the penalty. Returns `{"raged": int, "decayed": int}`. **Expiration:** `expire_quests_task` (Celery, 00:10 local) marks past-due active quests as expired. On completion: `_complete_quest` awards `coin_reward` (via `CoinService`), `xp_reward` (via `AwardService.grant`), and any `QuestRewardItem` entries (to `UserInventory`), then notifies. **Egg salvage** (2026-04-23, N1): when a `QuestRewardItem` is an `EGG` whose `pet_species` already appears in the user's `UserMount` table, the egg auto-salvages to `coin_value × quantity` instead of stacking dead eggs in inventory — same logic as cosmetic-dupe salvage in `DropService`. The `rewards["items"]` entry carries a `salvaged_to_coins` field so frontends can render the salvage path differently. Pinned by `QuestEggSalvageTest` in [`tests/test_services.py`](tests/test_services.py). `QuestParticipant` tracks per-user contribution — built multi-player-ready and now wired in the UI: parents can author **co-op quests** via the Trials parent challenge form (`/trials`) which exposes a co-op toggle + multi-child picker. The form posts a normal create, then the new endpoint **`POST /api/quests/{definition_id}/co-op/` `{coop_user_ids}`** (parent-only, family-scoped) wraps `QuestService.start_co_op_quest` to materialize one `Quest` row + N `QuestParticipant` rows in a single atomic call. `QuestWriteSerializer` accepts a `coop_user_ids` array (≥2) and a `skill_tags` array of `{skill_id, xp_weight}`; `CreateQuestView` consumes both atomically so untagged custom quests no longer silently award 0 skill XP. The form's **Advanced details** panel surfaces `allowed_triggers` and an `on_time_only` toggle (writes through to `trigger_filter.on_time`). **Reward rubric for new authors:** see the comment block at the top of [`content/rpg/initial/quests.yaml`](/content/rpg/initial/quests.yaml) for the Easy/Standard/Epic tier conventions; the 2026-04-23 review rebalanced Dragon Slayer (Standard tier), Focus Master, and Midnight Oil to align with the rubric.

- **Daily challenges** (added 2026-04-23): `DailyChallenge` is a lightweight once-per-day micro-quest **separate from the regular `Quest` slot**. Unique on `(user, date)` so every child gets exactly one per day. Five challenge types: `clock_hour`, `chores`, `habits`, `homework`, `milestone`. `DailyChallengeService.get_or_create_today` is idempotent (creates on first access, returns existing on re-call); `record_progress` is hooked into `GameLoopService.on_task_completed` so it advances alongside regular quest progress whenever a trigger maps to the challenge type (via `DAILY_CHALLENGE_TRIGGER_MAP`). `clock_hour` quantizes by `floor(duration_minutes / 60)` — a 30-minute clock-out doesn't advance a 1-hour challenge. `claim_reward` is idempotent: a second claim after the first no-ops (zero-out `coin_reward`/`xp_reward` after first claim as a marker), and posts the coin payout via `CoinLedger.Reason.DAILY_CHALLENGE` so an active Lucky Coin doubles it (the original ship used `adjustment`, which is excluded from `_BOOSTABLE_COIN_REASONS` — XP doubled but coins didn't; pinned by `test_claim_coins_are_boosted_by_active_lucky_coin` in [`tests/test_daily_challenge.py`](tests/test_daily_challenge.py)). Celery task `rotate_daily_challenges_task` (00:30 local) ensures every active child has today's row; it runs after `chronicle-chapter-transition` (00:25) and is safe to re-run within the same day. Endpoints: `GET /api/challenges/daily/` and `POST /api/challenges/daily/claim/`. Deliberately decoupled from `QuestDefinition` — the template list is a small enum (`DAILY_CHALLENGE_TEMPLATES` in `apps/quests/services.py`), not a CRUD'able catalog, so the daily slot stays small and doesn't compete with the real quest system.

## Tunables (constants in `apps/quests/services.py`)
- `TRIGGER_DAMAGE` — per-`TriggerType` boss damage table; clock_out multiplied by hours. QUEST_COMPLETE and PERFECT_DAY are intentionally absent (they're rewards, not damage sources).
- `RAGE_SHIELD_CLIMB_STEP = 15`, `RAGE_SHIELD_DECAY_STEP = 25`, `RAGE_SHIELD_CAP = 100` — asymmetric boss-quest rage tuning. Idle days climb +15 (saturates in ~7 idle days); active days decay -25. Decay outpaces climb on purpose so a child returning from a multi-day absence claws back faster than they fell behind — preserves the gentle-nudge doctrine. `RAGE_SHIELD_STEP` is kept as a back-compat alias pointing at the climb step.
