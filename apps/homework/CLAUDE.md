# apps/homework/

One-off homework assignments with a submit→approve workflow requiring proof image uploads. Homework pays **no money and no coins** — positioned as school duty, not work-for-hire. Rewards come from XP (via `HomeworkSkillTag`), RPG drops, streak credit, and quest progress.

## Models
- `HomeworkAssignment` (one-off task; `assigned_to` FK required; `due_date`; `effort_level` 1-5; optional `project` FK to AI-planned long-form).
- `HomeworkSubmission` (submit→approve with proof photos; inherits `ApprovalWorkflowModel`).
- `HomeworkProof` (required image uploads).
- `HomeworkTemplate` (reusable assignment configs).
- `HomeworkSkillTag` (XP distribution; `xp_amount` fixed-per-tag pattern).
- `HomeworkDailyCounter` (anti-farm gate; concrete `DailyCounterModel` subclass).

## Services
- `HomeworkService` — `create_assignment`, `submit_completion`, `approve_submission`, `plan_assignment` (AI), `get_timeliness`, `can_self_plan`.

## Endpoints
- `/api/homework/` (CRUD + `submit` + `save-template` + `plan`).
- `/api/homework-submissions/` (read-only + `approve` / `reject` / `withdraw`).
- `/api/homework-proofs/{id}/` (destroy-only — owner or parent — deletes a single proof image without voiding the submission).
- `/api/homework-templates/` (CRUD + `create-assignment`).
- `/api/homework/dashboard/`.

MCP tools: `list_homework`, `get_homework`, `create_homework`, `submit_homework`, `list_homework_submissions`, `approve_homework_submission`, `reject_homework_submission`.

## Gotchas

- **Homework**: one-off homework assignments with a submit-then-approve workflow requiring proof image uploads. Both parents and children can create assignments. **`assigned_to` is required on the model FK but optional on `HomeworkAssignmentWriteSerializer`** so two callers stay clean: children omit it (the viewset's `perform_create` auto-sets `assigned_to=request.user`), parents must pass `assigned_to: <child_id>` (the viewset raises `serializers.ValidationError` → 400 otherwise — without that guard the service hits a NOT NULL FK and 500s). The parent quick-action ("Create homework for a kid") navigates to `/quests?tab=study&new=1`; [`pages/Homework/index.jsx`](/frontend/src/pages/Homework/index.jsx) reads `?new=1` once on mount via `useSearchParams`, calls `openCreate()`, and clears the param so refresh doesn't re-trigger. The frontend [`HomeworkFormModal`](/frontend/src/pages/Homework/HomeworkFormModal.jsx) also short-circuits before submit when `isParent && !form.assigned_to` so a parent who skips the (HTML-required) child select still gets a friendly error instead of bouncing off the backend. **Homework pays no money and no coins** — it's positioned as school duty, not work-for-hire. Rewards come from XP (via `HomeworkSkillTag`), RPG drops, streak credit, and quest progress. `effort_level` is a 1-5 hint that scales the per-tag XP pool on approval via `EFFORT_XP_FACTOR` (1: 0.6x, 2: 0.8x, 3: 1.0x, 4: 1.4x, 5: 1.8x — default 3 keeps existing assignments at parity, 5 is roughly 3x the XP of 1). Children may set their own. The factor multiplies the pool BEFORE distribution so each tag's proportional split is preserved. Pinned by `EffortLevelXpScalingTests`. `HomeworkTemplate` stores reusable assignment configs (title/description/subject/effort_level/skill_tags). **On creation:** `HomeworkService.create_assignment` fires `GameLoopService.on_task_completed(child, "homework_created", ...)` ONLY for the first create per child per local day — subsequent creates skip the loop call entirely. The first-of-day check is gated on a `HomeworkDailyCounter` row (locked read-modify-write via `config.services.bump_daily_counter`); the counter row is independent of the assignment, so it survives both soft-delete (`is_active=False`) and hard-delete — a parent-cooperated `create → delete → create` cycle on the same day still skips the second loop call. This closes streak credit, drop, AND quest-progress farming through this trigger in one gate. Mirrors the same pattern used by `CreationDailyCounter` and `MovementDailyCounter`; all three concretes inherit from `config.base_models.DailyCounterModel`. Planner-style badges (`HOMEWORK_PLANNED_AHEAD`) are evaluated immediately on create regardless so logging early pays off fast. **On submission:** `submit_completion` records the `Timeliness` label (early/on_time/late/beyond_cutoff via `HomeworkService.get_timeliness`), attaches proof images, notifies parents — no AI calls, no snapshots. **On approval:** `approve_submission` distributes XP via skill tags, evaluates badges (including `HOMEWORK_ON_TIME_COUNT`), and fires `GameLoopService.on_task_completed(child, "homework_complete", {"on_time": bool})` so quests with `trigger_filter.on_time=true` can count only early/on-time submissions. **On delete:** `HomeworkAssignmentViewSet.perform_destroy` soft-deletes (`is_active=False`) so submission history is preserved. **RPG side effects from create are NOT reversed** — but the first-of-day gate makes the question moot for streaks/drops/quests; awarded XP/badges still don't unwind (matches every other approval flow). The anti-farm gate is pinned by `test_game_loop_fires_only_for_first_create_today`, `test_create_then_soft_delete_then_create_still_skips_second_loop`, and `test_create_then_hard_delete_then_create_still_skips_second_loop` in [`tests/test_services.py`](tests/test_services.py). **AI-planned long-form homework:** `HomeworkAssignment.project` (nullable FK to `Project`) links to a full project generated via Claude + MCP `create_project` tool use for complex multi-step assignments. `POST /api/homework/{id}/plan/` triggers AI planning. **Long-lead self-serve gate** (2026-04-25): parents can always plan; children can plan their own assignments only when the due date is at least `settings.HOMEWORK_SELF_PLAN_LEAD_DAYS` (default 3) days out. The "planning ahead" virtue earns autonomy; short-lead / urgent assignments stay parent-only because the conversation matters more than the AI plan when a child has waited too long. The gate is centralized in `HomeworkService.can_self_plan(user, assignment)` and surfaced to the frontend via the `can_plan` field on `HomeworkAssignmentSerializer` so the "Plan it out" button hides itself when self-serve isn't allowed. Routes: `/api/homework/` (CRUD + `submit` + `save-template` + `plan`), `/api/homework-submissions/` (read-only + `approve`/`reject`/`withdraw`), `/api/homework-proofs/{id}/` (destroy-only — owner or parent — deletes a single proof image without voiding the submission), `/api/homework-templates/` (CRUD + `create-assignment`), `/api/homework/dashboard/`. **Withdraw** (2026-05): owner-only `POST /api/homework-submissions/<id>/withdraw/` hard-deletes a pending submission **and** runs blob-first cleanup on every attached `HomeworkProof` image (Ceph DELETE before the row delete, matching the `ProjectPhotoViewSet.destroy` invariant). 400s once the submission has been approved. Frontend page: `/homework` (redirects to `/quests?tab=study`). Portfolio integration: approved homework proofs appear alongside project photos with filter tabs. MCP tools: `list_homework`, `get_homework`, `create_homework`, `submit_homework`, `list_homework_submissions`, `approve_homework_submission`, `reject_homework_submission`. **Scholar track content** (`content/rpg/initial/`): `Planner` / `Punctual` badges in `badges.yaml`, Scholar-themed cosmetics (`title-bookworm`, `title-scholar`, `frame-scholar`, `theme-library`) in `items.yaml`, and `Scholar's Week` / `On-Time Streak` / `Midnight Oil` quests in `quests.yaml`.

## Tunable settings
- `HOMEWORK_LATE_CUTOFF_DAYS` (default `3`) — past this many days late, a submission's `Timeliness` flips from `late` to `beyond_cutoff`. Homework pays no money/coins, so this only affects the submission label and the `on_time` quest filter — not any reward value.
- `HOMEWORK_SELF_PLAN_LEAD_DAYS` (default `3`) — how many days out an assignment's `due_date` must be for a child to self-trigger AI planning on their own assignments. Parents bypass this gate. Tuned so "planning ahead" earns autonomy while panic-mode (`due_date - today < N`) still requires a parent conversation. Read by `HomeworkService.can_self_plan`.
